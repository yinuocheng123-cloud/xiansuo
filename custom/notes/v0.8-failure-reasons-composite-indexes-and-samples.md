# 版本记录：v0.8-failure-reasons-composite-indexes-and-samples

## 1. 本轮目标

本轮目标是在 `v0.7-trends-and-indexes` 的基础上，继续把任务诊断、数据库查询支撑和平台规则沉淀向更接近商用后台的方向推进，重点完成三件事：

1. 给任务详情页增加“最近失败原因分类”
2. 继续优化 SQL 搜索与列表查询的组合索引策略
3. 继续沉淀地图、官网、公众号的规则命中样本库

为什么这三件事值得优先处理：

- 上一轮已经有“最近失败趋势”，但还不能快速看出失败集中在哪类问题上
- 结果页、线索库页、任务详情页都已经越来越依赖数据库查询，单列索引和 `pg_trgm` 只能解决一部分问题
- 平台规则已经拆出了适配器层，如果不把命中样本沉淀下来，后续继续补规则时仍然会依赖临场记忆

本轮目标归类：

- 功能补充
- 查询性能优化
- 可维护性提升

## 2. 背景与上下文

本轮承接记录：

- `custom/notes/v0.7-trends-and-indexes.md`

修改前系统状态：

- 任务详情页已经能展示最近失败趋势和最近成功入池趋势
- 结果页与线索库页的直接搜索已经下沉到 PostgreSQL `ILIKE`
- 数据库已有第一批 `pg_trgm` 和基础单列索引
- 地图、官网、公众号三类来源已有基础适配器，但命中样本主要体现在代码里，缺少独立沉淀

已知约束：

- 业务边界仍然严格限定在“高定木作公开商务线索”
- 不能扩展到隐私采集、登录后采集或泛行业线索系统
- 这一轮应坚持最小改动，避免把现有查询层和适配器层推翻重做

## 3. 问题分析

表面现象：

- 任务详情页能看到失败趋势，但审核或排障时仍然需要人工阅读每条错误文本，才能知道问题主要是限流、超时还是解析失败
- SQL 搜索虽然已经下沉，但高频页面的过滤条件和排序条件还缺少更有针对性的组合索引
- 平台规则虽然在增长，但缺少一份可以独立演进的样本库，后续规则增补容易重复劳动

核心矛盾：

- 当前系统已经从“能跑通”进入“持续迭代期”，这时最需要的不是再堆更多分散规则，而是补足诊断视角、查询支撑和可复用样本资产

主要原因拆分：

1. 失败诊断信息还是“明细优先”，没有把错误文本聚合到可判断的类别上
2. 列表查询真实使用路径已经固定，但索引结构还停留在基础兜底层
3. 平台规则样本没有脱离代码独立管理，导致经验沉淀效率偏低

## 4. 候选方案比较

### 方案 A：只补任务详情页，不动数据库和样本库

核心思路：

- 给任务详情页加一个失败原因分类面板
- 其他部分维持现状

优点：

- 页面层改动快，短时间内就能看到效果

缺点：

- 查询层短板继续保留
- 平台规则经验仍然散落在代码里
- 下一轮继续扩规则时上下文还是不完整

为什么放弃：

- 这轮三个问题本质上是同一阶段的配套工作，只补页面会留下明显断层

### 方案 B：同时补失败原因分类、组合索引和样本库

核心思路：

- 页面层补“最近失败原因分类”
- 数据库层补面向真实查询路径的组合索引
- 文档层新增独立样本库，并同步把样本增量回灌到平台适配器

优点：

- 诊断、性能和规则资产三条线一起前进
- 和 `v0.7` 已经形成的任务详情页、SQL 搜索、平台适配器结构天然兼容
- 后续继续补品牌归属和地址规则时不需要再从零整理样本

缺点：

- 需要同时修改页面、迁移、适配器、文档和版本记录
- 验证步骤会比单点页面改动更多

为什么选中：

- 这是当前阶段收益最高、且不会破坏现有架构一致性的做法

## 5. 最终决策

最终采用方案 B，并落实为以下设计决策：

1. 失败原因分类采用轻量规则法，直接基于失败样本与重试记录里的错误文本聚合
2. 组合索引优先围绕真实查询路径，而不是一次性设计很多理论上的索引
3. 规则命中样本库单独落到 `custom/rule-samples/`，作为后续继续补规则的正式资产
4. 平台适配器只做小步增量，继续沿用当前“地图 / 官网 / 公众号”分层，不做重写

这样设计的原因：

- 失败分类本质上是运维视角的快速判断，不需要引入复杂分类系统
- 组合索引最怕过多过宽，因此应该只覆盖当前高频路径
- 样本库如果继续混在代码中，后续复盘和交接都会继续变难

## 6. 具体实现

本轮修改文件如下：

- `src/lib/render.js`
  增加失败原因归类函数、失败原因聚合逻辑，以及任务详情页“最近失败原因分类”面板；同时在失败样本和重试记录明细里显示原因分类列
- `database/migrations/004_lead_composite_indexes.sql`
  新增第二批组合索引，覆盖任务详情、结果页、线索库的高频筛选与排序路径
- `database/postgres.sql`
  同步更新数据库结构快照，纳入本轮新增组合索引
- `src/lib/providers/pageAdapters.js`
  继续补地图、官网、公众号中与品牌归属、地址有关的标签型命中样本
- `custom/rule-samples/platform-hit-samples.md`
  新增独立规则命中样本库，沉淀地图、官网、公众号的品牌归属与地址样本
- `README.md`
  更新任务详情说明、迁移说明、组合索引说明和样本库路径
- `AGENTS.md`
  更新最新一期正式记录路径

组合索引的具体策略：

- `task_id + match_score desc + created_at desc`
  对应任务详情页最近线索列表
- `review_status + match_score desc + created_at desc`
  对应结果页与线索库页最常见的排序路径
- `province + city + match_score desc + created_at desc`
  对应省市筛选
- `source_platform + lead_type + match_score desc + created_at desc`
  对应来源平台和线索类型组合筛选
- `brand + match_score desc + created_at desc`
  对应品牌筛选，且使用部分索引避免大量空品牌值拖累

## 7. 本轮优点

1. 任务详情页从“看见失败趋势”进一步推进到“看懂失败主要集中在哪类问题”
2. 数据库索引策略开始和真实查询路径一一对应，而不是只靠通用兜底索引
3. 平台规则样本首次作为独立资产沉淀下来，后续规则演进可以直接复用
4. 平台适配器的样本补充仍然保持增量方式，没有破坏现有分层

## 8. 本轮缺点与代价

1. 失败原因分类目前仍然是规则法聚合，不是完整的失败事件体系，复杂错误文本仍可能被归到“其他异常”
2. 组合索引虽然更贴近当前查询路径，但数据量继续扩大后，仍可能需要进一步做执行计划复核
3. 规则样本库目前还是人工维护文档，不是自动采集的样本仓库
4. 当前直接搜索底层仍然是 `ILIKE + pg_trgm`，如果未来搜索语义继续变复杂，还需要评估全文检索方案

若后续验证中出现问题，应优先关注：

- `render.js` 里新增原因分类逻辑是否影响原有任务详情渲染
- 新增索引名称是否与已有索引冲突
- `pageAdapters.js` 的正则样本补充是否引入过宽匹配

## 10. 本轮结论

本轮最重要的结论是：

- 当前系统已经不只是继续“补规则”，而是在开始形成更完整的诊断面板、数据库查询支撑和规则知识资产

被进一步证实的假设：

- 任务页与任务详情页的诊断价值，随着聚合视角增强，会明显提升排障效率
- SQL 搜索一旦下沉数据库层，索引策略就必须继续按真实使用路径演进
- 平台规则相关经验单独沉淀到样本库，是后续稳定迭代的必要条件
