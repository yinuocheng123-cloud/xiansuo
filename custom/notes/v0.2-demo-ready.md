# 版本记录：v0.2-demo-ready

## 1. 本轮目标

本轮目标是在 `v0.1-mvp-baseline` 的基础上，把系统从“能运行的骨架”推进到“可直接演示的 MVP”。

为什么值得优先做：

- 上一轮已经有完整骨架，但真实抓取依赖公网环境，演示时不稳定
- 如果没有稳定 demo，用户很难快速判断后台流是否符合预期
- 演示层做得好，可以减少后续对“系统到底长什么样”的反复沟通

本轮目标归类：

- 可维护性提升
- 演示可用性提升
- 体验补齐

## 2. 背景与上下文

本轮承接记录：

- `custom/notes/v0.1-mvp-baseline.md`

本次与上一轮关系：

- 属于延续，而不是推翻
- 延续了“零依赖、规则优先、边界收敛”的总体方案
- 修正了上一轮“只能靠真实抓取看效果”的演示不足

当前系统在修改前的状态：

- 可启动
- 可创建任务
- 可进入审核和导出
- 但若公网抓取不可用，页面容易缺少足够可展示内容

## 3. 问题分析

表面现象：

- 系统能跑，但 demo 体验依赖外网和搜索结果稳定性
- 用户问“做完后能否看到 demo”，说明当前最关键的不只是代码存在，而是能否直观看到产品效果

核心矛盾：

- 真正业务流需要抓取公开网页
- 但演示阶段又不能把可见性完全绑死在外部环境上

主要原因：

1. 候选页发现依赖公网搜索结果
2. 演示阶段通常比真实生产更强调可重复性
3. 如果没有本地样例，很多页面在空数据状态下不够直观

## 4. 候选方案比较

### 方案 A：继续只依赖真实抓取演示

优点：

- 演示更接近真实运行

缺点：

- 非常依赖网络环境
- 不稳定
- 不利于快速确认页面效果

为什么未选：

- 不满足“现在就能看 demo”的诉求

### 方案 B：增加本地演示数据与一键装载入口

优点：

- 可重复
- 不依赖外部网页稳定性
- 便于演示审核、筛选、导出全流程

缺点：

- 演示数据不是真实抓取结果
- 需要额外维护一份样例数据

为什么选中：

- 最适合当前阶段
- 不改变业务边界，也不影响真实抓取链路

## 5. 最终决策

最终采用方案 B，并做以下设计：

1. 增加本地演示数据模块
2. 提供 `npm run demo:seed`
3. 在任务页增加“加载演示数据”入口
4. 演示数据严格只覆盖高定木作公开商务线索，不混入泛行业样例

延续的旧思路：

- 继续使用本地 JSON 存储
- 继续使用规则法字段与评分体系

本轮修正的旧判断：

- 上一轮默认“可运行就够了”，本轮补充为“可演示也要稳定”

## 7. 本轮优点

- 现在可以稳定看到 demo，不依赖真实抓取是否即时成功
- 页面更像一个可展示产品，而不是纯开发骨架
- 演示数据仍然遵守业务边界，没有扩成泛行业样例
- 新增演示能力没有打散原有模块边界

## 8. 本轮缺点与代价

1. 演示数据需要额外维护
2. 用户看到的 demo 不等于真实线上抓取结果
3. 如果后续字段变化，演示数据也要同步调整

当前接受的妥协：

- 为了稳定演示，接受本地样例和真实抓取并存

## 9. 验证与结果

本轮计划验证：

- `npm run check`
- `npm run demo:seed`
- 本地启动服务并验证任务页、结果页、线索库页
- 验证演示数据写入后页面可见

已确认结论：

- 演示入口和演示脚本已补齐

## 10. 本轮结论

本轮最重要的结论：

- 对当前阶段而言，“可稳定演示”是 MVP 体验的重要组成部分，不能完全依赖真实抓取链路临场表现

新形成的经验：

- 后续做来源适配和抽取增强时，也要尽量保留可重复的本地演示能力

## 11. 下一轮建议

1. 为真实抓取结果补更细的字段质量提示
2. 引入 PostgreSQL 持久化替代 JSON
3. 为地图、公众号、官网分别补独立适配器
