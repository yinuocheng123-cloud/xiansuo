# 版本记录：v0.9-failure-reason-insights-and-explain

## 1. 本轮目标

本轮目标是在 `v0.8-failure-reasons-composite-indexes-and-samples` 的基础上，继续沿着“任务诊断可读性 + 数据库查询可验证性 + 抽取规则精细化”这条线推进，重点完成三件事：

1. 给任务详情页补“失败原因 Top 摘要”和按原因的最近变化提示
2. 用 `EXPLAIN` 继续复核组合索引是否命中预期查询
3. 继续把样本库里的高价值样本回灌到品牌清洗和地址清洗规则里

为什么优先做这三件事：

- 上一轮已经能看到失败原因分类，但还缺少一眼可读的摘要和变化判断
- 组合索引已经补了两轮，如果不做 `EXPLAIN`，就无法判断“索引是否真的在用”
- 平台规则样本库已经沉淀下来，最自然的下一步就是把高价值样本继续回灌到清洗规则，而不是让样本库停留在文档层

本轮目标归类：

- 功能补充
- 可维护性提升
- 查询验证与优化

## 2. 背景与上下文

本轮承接记录：

- `custom/notes/v0.8-failure-reasons-composite-indexes-and-samples.md`
- `custom/notes/v0.7-trends-and-indexes.md`

修改前系统状态：

- 任务详情页已经能展示最近失败趋势、最近成功入池趋势和失败原因分类
- `database/migrations/004_lead_composite_indexes.sql` 已经为任务详情、结果页和线索库补了组合索引
- 平台规则命中样本库已单独沉淀到 `custom/rule-samples/platform-hit-samples.md`
- 抽取层中品牌清洗和地址清洗仍主要覆盖第一批标签样本

延续的旧思路：

- 仍然坚持最小改动，不去重做页面结构和查询层
- 仍然只围绕“高定木作公开商务线索”这一条业务边界开发
- 仍然优先用规则法和配置化思路推进，而不是引入复杂 AI 主导流程

## 3. 问题分析

表面现象：

- “最近失败原因分类”虽然能看出分布，但不能快速回答“最近主要是哪种失败在变多”
- 已经新增的组合索引如果没有 `EXPLAIN` 验证，后续容易继续凭经验加索引
- 样本库里的“联合品牌”“品牌方”“总部地址”“接待中心”等样本还没有完整回灌到清洗规则

核心矛盾：

- 系统已经进入持续打磨阶段，单纯“补功能”已经不够，需要把可读性、可验证性和规则沉淀打通

主要原因：

1. 失败原因页面仍偏分布展示，缺少摘要层和趋势解释层
2. SQL 层虽然已经有索引，但尚未逐条核验真实查询路径
3. 品牌和地址清洗规则对边界样本的兼容性还不够细

## 4. 候选方案比较

### 方案 A：只补任务详情摘要，不做 `EXPLAIN`

核心思路：

- 只在任务详情页补 Top 摘要和变化提示
- 索引效果继续按经验判断

优点：

- 页面可见性提升快

缺点：

- 无法确认索引是否真实命中
- 后续查询优化依然缺少事实依据

为什么放弃：

- 这一轮用户明确要求继续用 `EXPLAIN` 复核，不能只停在页面层

### 方案 B：页面摘要、执行计划复核、规则回灌一起做

核心思路：

- 页面层补摘要和变化提示
- 数据库层用 `EXPLAIN` 检查关键查询路径
- 规则层继续吸收样本库中的高价值标签

优点：

- 诊断、查询和抽取三条线能形成闭环
- 能把“感觉这样更好”变成“验证过这样更好”
- 和 `v0.8` 的结构天然兼容

缺点：

- 本轮需要同时修改页面、查询逻辑、文档和正式记录

为什么选中：

- 这是对上一轮最自然也最稳妥的延续

## 5. 最终决策

最终采用方案 B，并形成以下决策：

1. 失败原因 Top 摘要采用轻量聚合，不引入新的事件表或统计表
2. 最近变化提示使用“最近窗口 vs 上一阶段同窗口”的方式，避免复杂时间序列设计
3. `EXPLAIN` 优先覆盖当前最核心的 5 条查询路径
4. 品牌模糊筛选优先优化查询写法，让已有 trigram 索引有机会参与，而不是先急着再加新索引
5. 样本回灌继续落在 `extractor.js`，不新增新的清洗模块

## 7. 本轮优点

1. 任务详情页从“看分布”进一步推进到“看重点和变化”
2. 组合索引不再只是“建过了”，而是有 `EXPLAIN` 结果支撑
3. 品牌模糊筛选的 SQL 写法变得更有利于 trigram 索引参与
4. 样本库里的高价值边界样本已经继续回灌到抽取清洗逻辑

## 8. 本轮缺点与代价

1. 失败原因变化提示仍然是轻量窗口比较，不是完整的时间序列统计
2. 品牌模糊筛选在 demo 小样本数据下，默认计划仍可能因为数据量太小而偏向顺序扫描
3. 样本回灌仍然主要靠人工筛选与人工总结，尚未形成自动沉淀机制
4. 本轮没有继续新增数据库结构，只是在已有结构上做验证和查询写法修正

## 10. 本轮结论

本轮最重要的结论是：

- 当前系统的下一步优化，不是继续无差别加索引，而是通过 `EXPLAIN` 逐条验证关键路径，并在必要时优先优化查询写法

被进一步证实的假设：

- 失败原因面板如果增加摘要层和变化层，排障可读性会明显提升
- 组合索引中任务详情、审核状态、省市和来源+类型路径已经达到预期
- 品牌模糊筛选的关键问题不是“没有索引”，而是“查询写法是否给索引参与机会”
