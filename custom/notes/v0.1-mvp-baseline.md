# 版本记录：v0.1-mvp-baseline

## 1. 本轮目标

本轮目标是从 0 搭建“高定木作公开商务线索采集系统”的首个可运行 MVP，优先把最核心的采集、识别、审核和导出流程跑通。

为什么这个问题值得优先处理：

- 当前仓库为空，没有基线工程，也没有可继承实现
- 用户需求边界很清晰，适合先做规则法 MVP，建立后续迭代基础
- 如果首轮没有把边界、字段和审核流定住，后续很容易偏离到泛行业采集或隐私采集方向

本轮目标归类：

- 功能补充
- 结构建立
- 可维护性提升

## 2. 背景与上下文

本轮修改前状态：

- 仓库为空目录
- 未发现现有 `AGENTS.md`
- 未发现 `custom/notes/` 历史记录
- 未发现 `custom/notes/experiments/` 实验记录

因此，本轮不属于承接已有代码的增量修改，而是首轮基线建立。

已知约束：

- 只围绕“高定木作公开商务线索”开发
- 只采集公开网页中的公开商务信息
- 不能扩展成泛行业线索采集系统
- 不能扩展成隐私信息采集工具
- MVP 业务主字段严格围绕 10 个核心字段
- 评分规则、关键词规则、平台规则必须可配置

## 3. 问题分析

表面问题：

- 需要一个 Web 后台系统
- 需要任务、结果、审核、线索库 4 个页面
- 需要抓取、提取、评分、去重和导出

核心矛盾：

- 用户要求“可运行”和“直接进入开发输出”，但仓库当前没有任何代码和依赖基础
- 如果一开始引入较重框架，会在空仓库里带来大量依赖安装、构建和环境不确定性
- 如果完全只写设计文档，又达不到“第一版可运行代码骨架”的目标

主要原因：

1. 当前是空仓库，没有现成工程约束可跟随
2. 本地运行环境未知，过重技术栈会降低首轮可运行性
3. 业务边界很强，必须在架构上避免“以后方便”导致当前就做成泛平台

次要表象：

- 数据库尚未就位
- 真实搜索抓取存在来源稳定性问题
- 没有现成队列系统

## 4. 候选方案比较

### 方案 A：Next.js + PostgreSQL + Prisma

核心思路：

- 采用较常见的全栈技术栈，前后端统一在一个项目中实现

优点：

- 后续扩展方便
- 社区成熟
- 页面能力完整

缺点：

- 首轮需要安装较多依赖
- 空仓库里第一次落地成本较高
- 当前环境下不一定能立即完成依赖安装与数据库初始化

风险：

- 首轮交付可能停留在脚手架层面，未必能快速跑通完整流程

为什么本轮未选：

- 本轮优先目标是“尽快建立可运行基线”，不是一次性确定最终重栈

### 方案 B：Node 原生 HTTP + 服务端渲染 + 本地 JSON 存储

核心思路：

- 使用 Node 20 原生能力，尽量不引入外部依赖
- 用 JSON 存储替代数据库作为运行基线，同时补齐 PostgreSQL 表结构文档

优点：

- 可运行性最高
- 代码路径短，便于快速理解和继续迭代
- 规则层、抓取层、渲染层边界清晰
- 后续迁移到 PostgreSQL 时，业务逻辑基本可复用

缺点：

- 当前不是最终生产型技术栈
- UI 能力和工程化能力不如前端框架
- 并发、任务队列和可观测性较基础

风险：

- 搜索结果抓取策略后续可能要替换
- JSON 存储不适合多人并发与大规模数据量

为什么本轮选中：

- 最符合“先做最小可运行版本”的要求
- 能在空仓库中最快建立完整业务闭环

## 5. 最终决策

最终采用方案 B：

- 服务端：Node 原生 `http`
- 页面：服务端渲染 HTML
- 存储：本地 JSON 文件
- 规则：关键词、平台、评分集中配置
- 抓取：Bing 公开搜索结果页发现候选，再抓取候选页面

关键设计决策：

1. 业务主字段严格维持 10 个
2. 为审核与去重，仅新增少量支撑字段，如 `raw_summary` 与 `duplicate_state`
3. 不做复杂任务队列，先用进程内异步执行器
4. 不做 AI 主导抽取，先建立稳定规则基线
5. 所有边界判断集中在规则层，防止后续开发时悄悄泛化业务范围

沿用旧逻辑情况：

- 无。因为仓库此前没有实现。

新增约束：

- 未识别到高价值信号且分数低于 60 的结果直接标记为 `已丢弃`
- 去重只做“疑似重复标记”，不直接删除

## 6. 具体实现

本轮新增与改动文件如下：

- `AGENTS.md`
  - 建立仓库级代理约束摘要，并写入最新一期记录路径
- `README.md`
  - 汇总技术方案、表设计、API、页面结构与运行方式
- `database/postgres.sql`
  - 提供 `tasks` 和 `leads` 的 PostgreSQL 建表语句
- `public/styles.css`
  - 提供 4 个后台页面的基础样式
- `src/config/keywords.js`
  - 维护木作关键词、排除词和城市词
- `src/config/platforms.js`
  - 维护来源平台识别规则
- `src/config/scoring.js`
  - 维护评分规则和阈值
- `src/lib/storage.js`
  - 本地 JSON 存储
- `src/lib/html.js`
  - HTML 清洗与摘要提取
- `src/lib/http.js`
  - 有限速、超时和重试的抓取工具
- `src/lib/rules.js`
  - 关键词匹配、线索类型判定、平台识别
- `src/lib/scoring.js`
  - 匹配分计算
- `src/lib/dedupe.js`
  - 基础去重标记
- `src/lib/extractor.js`
  - 名称、品牌、城市、联系方式、地址提取
- `src/lib/providers/searchProvider.js`
  - 候选页发现
- `src/lib/csv.js`
  - 已保留线索导出
- `src/lib/taskRunner.js`
  - 任务创建、执行、线索入库、审核更新
- `src/lib/render.js`
  - 任务页、结果页、审核页、线索库页渲染
- `src/server.js`
  - 页面路由和 JSON API 入口

模块关系：

- `server.js` 负责路由与响应
- `taskRunner.js` 负责业务流编排
- `providers/searchProvider.js` 与 `http.js` 负责候选页抓取
- `extractor.js`、`rules.js`、`scoring.js` 负责抽取判断
- `render.js` 负责页面输出

## 7. 本轮优点

相比空仓库状态，本轮带来的正向效果：

- 已建立可运行的 Web MVP 基线
- 已将核心业务边界显式固化进规则配置
- 已实现任务、结果、审核、导出的完整闭环
- 已实现基础评分与疑似去重
- 已为后续数据库迁移准备 SQL 结构
- 已把思路档案补齐，后续代理无需重新猜测为什么这样搭

改善的质量属性：

- 可读性：模块职责明确
- 可维护性：规则集中配置
- 可扩展性：抓取、规则、存储均可独立替换
- 调试便利性：不依赖复杂构建链路

## 8. 本轮缺点与代价

这一部分必须真实保留 trade-off：

1. 当前抓取源以 Bing 搜索结果页为主，稳定性有限
2. JSON 存储只适合 MVP，不适合真正多用户并发
3. 抽取逻辑仍是规则法，复杂品牌归属和地址解析还不够精确
4. 当前任务执行器是进程内队列，服务重启会中断运行中的任务
5. 没有做更深入的平台适配器，如公众号正文结构化解析、地图结果标准化解析

当前接受的妥协：

- 先用最稳妥的零依赖方案建立闭环，再逐步替换关键薄弱点

未来可能推翻重做的部分：

- 存储层
- 搜索候选发现实现
- 更细的去重算法

## 9. 验证与结果

本轮已做验证：

- 使用 Node 20 环境检查运行时可用
- 对新增代码执行 `node --check src/server.js`
- 启动服务进行短时自检，确认服务可监听端口

已确认结论：

- 项目能够在零依赖条件下启动 Web 服务
- 页面路由与 API 路由已接通
- 任务创建、结果查询、审核更新、CSV 导出代码路径已完整落地

尚未完成的验证：

- 未对真实公网抓取做端到端验证

原因：

- 当前开发阶段优先建立可运行结构；真实抓取依赖外部网络环境和目标页面可访问性

剩余风险：

- 真实线上页面结构差异可能影响抽取精度
- 搜索来源可能因为页面结构变化而需要调整解析规则

## 10. 本轮结论

本轮最重要的结论：

- 在业务边界明确、仓库为空的前提下，先建立一个“零依赖、规则优先、闭环完整”的 MVP 基线，是当前最稳妥的推进方式

被证实的假设：

- 用原生 Node 方案可以快速搭出完整业务骨架
- 评分规则、平台规则、关键词规则集中配置后，后续迭代会更安全

被否定的假设：

- “首轮必须直接上完整前端框架和数据库”并不是必要条件

后续默认应继承的经验：

- 继续坚持业务边界收敛
- 继续坚持规则集中配置
- 继续保留详细思路档案

## 11. 下一轮建议

最值得继续推进的 3 件事：

1. 把 JSON 存储替换为 PostgreSQL，并补上初始化脚本
2. 把候选页发现从单一搜索结果解析升级为多来源适配器
3. 提升抽取精度，补充品牌识别、地址识别和名称清洗规则

收益最高：

- PostgreSQL 持久化替换

风险最低：

- 品牌识别与地址提取规则增强

需要更多信息后再做：

- 是否接入 AI 作为二次判定辅助，而不是替代规则基线
