# 版本记录：v0.7-trends-and-indexes

## 1. 本轮目标

本轮目标是在 `v0.6-task-detail-sql-search` 的基础上，继续把系统往更接近商用后台的方向修订，重点解决三件事：

1. 给任务详情页补“最近失败趋势 / 最近成功入池趋势”
2. 给 SQL 搜索补索引优化
3. 继续按平台补更细的品牌归属和地址规则样本

为什么这轮值得优先做：

- 上一轮虽然已经有任务详情页，但诊断信息仍偏“明细展开”，趋势视角还不够
- SQL 搜索已经下沉到数据库层，如果不马上补索引，后面数据量上来会先拖慢体验
- 品牌归属和地址完整度仍然是当前审核效率的关键瓶颈

本轮目标归类：

- 功能补充
- 查询性能优化
- 提取质量提升

## 2. 背景与上下文

本轮承接记录：

- `custom/notes/v0.6-task-detail-sql-search.md`

本次与上一轮的关系：

- 完全延续上一轮“任务详情 + SQL 查询 + 提取规则增强”的方向
- 不是做新系统，而是继续把已经形成的后台能力打磨得更像可持续使用的产品

修改前状态：

- 任务详情页已经能看失败样本、重试记录和当前候选页状态
- 结果页直接搜索已经下沉到 SQL
- 但任务详情页还缺趋势视角
- SQL 查询尚未专门配套搜索索引
- 解析器已增强，但平台样本还不够细

## 3. 问题分析

表面现象：

- 任务详情页能看“发生了什么”，但还不容易看出“最近是在变好还是变坏”
- 搜索下沉到 SQL 后，性能路径已经清晰，索引如果不跟上就会留下明显短板
- 规则虽然越来越多，但如果平台级品牌/地址样本不继续补，准确率提升会变慢

核心矛盾：

- 当前系统已经开始具有“商用后台”的雏形，但还缺少趋势可视化和数据库层面的性能兜底

主要原因：

1. 详情页只有明细，没有聚合趋势
2. SQL 查询依赖 `ILIKE`，缺少对应索引
3. 地图、官网、公众号的品牌/地址标签样本仍可以继续细化

## 4. 候选方案比较

### 方案 A：只补趋势面板，不动索引和样本规则

优点：

- 页面见效快

缺点：

- 查询性能问题继续留存
- 解析器精度问题继续留存

为什么放弃：

- 这轮三件事是互相支撑的，只做趋势面板会留下明显短板

### 方案 B：趋势、索引和样本规则一起补

优点：

- 页面可见性、数据库性能和提取质量同时提升
- 与上一轮的 SQL 搜索改造形成闭环

缺点：

- 需要同步修改页面、数据库和演示数据

为什么选中：

- 这是当前阶段最符合“持续修订到商用级”的推进方式

## 5. 最终决策

最终采用方案 B，并落实为以下动作：

1. 在任务详情页中加入最近失败趋势和最近成功入池趋势
2. 补 `pg_trgm` 扩展和一组与 SQL 搜索相关的索引
3. 继续为地图、官网、公众号补品牌和地址的标签型样本规则
4. 更新 demo 时间样本，让趋势面板在演示环境下也能真实展示

## 7. 本轮优点

1. 任务详情页不再只有明细，还能看最近趋势
2. SQL 搜索有了更像样的数据库层支撑
3. demo 不再只是静态样例，而是能真实展示趋势变化
4. 平台级品牌归属和地址规则样本更丰富

## 8. 本轮缺点与代价

1. 当前失败趋势仍基于失败样本而不是完整失败事件流，精度受样本保留数量影响
2. `pg_trgm` 索引适合当前 MVP，但后续还可能需要更系统的全文检索方案
3. 趋势面板目前是后端渲染的轻量列表，不是完整图表系统

- 问题不是代码缺陷，而是执行顺序错误
- 之后串行重跑 `demo:seed` 已恢复正常

## 10. 本轮结论

本轮最重要的结论是：

- 当前系统已经从“有任务详情页”推进到“有任务趋势视角、SQL 搜索索引和更细平台规则样本”的阶段

被证实的假设：

- 趋势面板即使不用复杂图表，也能明显提升任务排障可读性
- SQL 搜索下沉后，索引补齐是非常必要且收益直接的一步
- demo 时间样本必须跟页面演进同步，否则很多增强无法被真实演示出来

## 11. 下一轮建议

1. 给任务详情页增加“最近失败原因分类”
2. 给 SQL 搜索补更细的组合索引或全文检索预案
3. 继续沉淀地图、官网、公众号的规则命中样本库
