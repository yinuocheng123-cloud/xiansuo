# 版本记录：v0.4-postgres-adapters

## 1. 本轮目标

本轮目标是在 `v0.3-review-quality` 的基础上，完成三项更偏底层和真实可用性的改造：

1. 把 JSON 存储正式切到 PostgreSQL
2. 给地图、官网、公众号分别补更细的来源适配
3. 提升品牌和地址提取精度

为什么这轮值得优先做：

- 前几轮已经有可展示 demo 和审核体验
- 但只要底层还是 JSON，本项目就还停留在“演示骨架”阶段
- 如果来源适配和字段提取精度不增强，真实抓取的价值会很快碰到天花板

本轮目标归类：

- 结构升级
- 可维护性提升
- 真实可用性提升

## 2. 背景与上下文

本轮承接记录：

- `custom/notes/v0.3-review-quality.md`

本次与上一轮的关系：

- 延续上一轮“不要先扩功能，先把已有能力做扎实”的判断
- 这次把重点从页面呈现层转到底层存储和抽取精度

修改前状态：

- 页面和 demo 已可展示
- 但存储仍是 JSON
- 来源适配还主要靠统一规则
- 品牌和地址识别偏粗糙

## 3. 问题分析

表面现象：

- 系统能用，但不够像一个可以继续长大的真实后台

核心矛盾：

- 用户现在已经不只是要看 demo，而是要继续往“真实可用的采集系统”推进
- 如果底层存储和来源适配不升级，后续越改越会受限

主要原因：

1. JSON 读写不适合持续扩展
2. 地图、官网、公众号的页面结构差异明显
3. 品牌和地址提取不能只靠一套统一正则

## 4. 候选方案比较

### 方案 A：继续保留 JSON，先补更多页面功能

优点：

- 页面上见效快

缺点：

- 底层债务继续积累
- 后续再切库成本会更高

为什么未选：

- 不符合当前阶段最重要的技术路径

### 方案 B：先做 PostgreSQL + 来源适配 + 抽取增强

优点：

- 更接近真实可用系统
- 后续新增来源和规则时不需要先推翻存储层
- 能直接提升真实抓取的字段质量

缺点：

- 改动面比纯页面修订更大
- 需要同步改脚本、seed、服务读写链路

为什么选中：

- 这是把项目从“演示级 MVP”推进到“可持续迭代 MVP”的关键一步

## 5. 最终决策

最终采用方案 B，并落实为以下动作：

1. 用 PostgreSQL 替换原有 JSON 存储
2. 增加数据库初始化脚本
3. 演示数据改为写入 PostgreSQL
4. 给地图、官网、公众号增加来源适配器
5. 品牌识别增加站点名、公众号名、域名与名称前缀综合判断
6. 地址识别增加适配器候选和多层回退规则

延续的旧思路：

- 继续保持 10 个核心业务字段不扩张
- 继续坚持规则优先
- 继续保持业务边界只围绕高定木作公开商务线索

修正的旧判断：

- 之前默认“数据库可以后面再说”，这轮明确把它提前到了当前阶段

## 7. 本轮优点

- 存储层已正式切换为 PostgreSQL
- demo 不再只是文件写入，而是走真实数据库链路
- 来源适配更清晰，后续扩更多平台时结构更好接
- 品牌和地址识别比之前更接近真实页面情况
- 任务页开始显示最近错误，便于排查抓取问题

## 8. 本轮缺点与代价

1. 代码复杂度明显高于 JSON 版本
2. 当前默认数据库连接串仍是本地开发配置，生产化前需要更严格管理
3. 搜索候选页发现仍然主要依赖 Bing 公开搜索结果页
4. 适配器已经拆分，但还不算完整平台级解析器

当前接受的妥协：

- 先把最关键的地图、官网、公众号适配拆出来
- 先建立 PostgreSQL 基线，再继续做更细的迁移和监控

- 后续每次继续增强抓取和审核体验时，都应该默认以数据库持久化和来源适配结构为前提，而不是回到文件存储思路。
