# 版本记录：v0.5-province-progress-migrations

## 1. 本轮目标

本轮目标是在 `v0.4-postgres-adapters` 的基础上，继续把系统从“PostgreSQL 可跑通”推进到“任务与筛选能力更贴近真实使用”的阶段，重点解决四类问题：

1. 增加省级范围任务与省级筛选
2. 将任务计划数量上限放开到 `5000`
3. 给任务页增加最近抓取进度、失败链接样本和重试记录
4. 继续细化地图、官网、公众号解析器，并补齐数据库迁移脚本与环境变量说明

为什么这轮值得优先做：

- 上一轮已经把存储切到了 PostgreSQL，但任务维度仍偏“城市级 demo”
- 真实线索收集场景里，省级搜集和大批量任务是非常常见的
- 如果没有任务级进度和失败诊断，任务一旦失败就只能靠猜
- 数据库虽然已经接入，但如果迁移与环境说明不补齐，后续接手成本会很高

本轮目标归类：

- 功能补充
- 可维护性提升
- 可观察性提升

## 2. 背景与上下文

本轮承接记录：

- `custom/notes/v0.4-postgres-adapters.md`

本次与上一轮的关系：

- 延续上一轮“先把底层基线做稳”的思路
- 不是推翻 PostgreSQL 基线，而是在这个基线上继续补任务粒度、运行可见性和文档完备度

修改前系统状态：

- 已有 PostgreSQL 持久化
- 已拆出地图、官网、公众号适配器
- 结果页与审核页已经可以演示
- 但任务仍然主要围绕城市范围和小规模 demo
- 任务执行中缺少可持续追踪的诊断信息
- 迁移机制虽已具备，但说明文档还不够完整

已知约束：

- 系统必须继续严格服务“高定木作公开商务线索”
- 不允许扩成泛行业系统或隐私信息采集工具
- 评分、关键词、来源平台规则仍应保持集中配置
- 继续坚持最小改动，不引入重型框架

## 3. 问题分析

表面现象：

- 用户已经明确提出希望按“省”直接搜索，并把任务规模放开到 `5000`
- 现有任务页虽然能看状态，但无法快速判断“卡在哪”“失败了什么”“是否重试过”
- 文档里对数据库迁移和环境变量的说明不够明确，后续容易出现“代码能跑，但别人接不上”的问题

核心矛盾：

- 当前系统已经不再只是展示型 demo，而是需要逐步承担更接近真实采集任务的职责
- 一旦任务范围变大，如果没有进度与失败记录，可用性会迅速下降

主要原因：

1. 任务模型中缺少省级维度与任务诊断字段
2. 页面筛选能力仍以城市维度为主
3. 搜索候选页发现逻辑没有充分表达“省 + 市 + 关键词”的组合范围
4. 来源解析器虽已拆分，但品牌、地址、电话的取值仍缺少更细的候选来源
5. 数据库迁移和环境说明没有形成接手友好的显式文档

次要原因：

- README 仍保留部分旧阶段表达，和当前实现不完全对齐

## 4. 候选方案比较

### 方案 A：只做前端筛选补丁，不改数据库和任务模型

核心思路：

- 页面上新增“省份”输入框和“直接搜索”输入框
- 不增加任务级进度、失败样本和重试记录

优点：

- 改动少
- 页面可以很快看到新筛选项

缺点：

- 任务真实运行状态仍不可见
- 后端没有省级维度，前端筛选也只是表面补丁
- 无法满足用户关于“最近抓取进度、失败样本、重试记录”的明确要求

为什么放弃：

- 这会造成“页面看起来支持了，但数据模型没有真的跟上”的问题

### 方案 B：同时补数据库字段、任务执行记录、筛选能力和文档

核心思路：

- 在 PostgreSQL 基线上新增省级维度和任务诊断字段
- 搜索候选页发现逻辑支持省、市两级范围
- 页面层直接展示任务进度、失败样本和重试记录
- README 与迁移脚本同步补齐

优点：

- 用户需求一次性贯通
- 页面展示、数据库结构、任务执行链路保持一致
- 后续继续迭代解析器和调度器时不需要再回头补基础字段

缺点：

- 改动会跨越存储、执行器、页面、文档多个文件
- 代码复杂度略有上升

为什么选中：

- 这是当前阶段最划算的改动，能同时提升真实可用性和后续可接手性

## 5. 最终决策

最终采用方案 B，并落实为以下动作：

1. 在 `tasks` 中补齐 `province`、`latest_progress`、`failure_samples`、`retry_records`
2. 在 `leads` 中增加 `province`
3. 任务创建允许“省或市至少填一项”，计划数量上限提高到 `5000`
4. 搜索候选页发现逻辑按“省 + 市 + 关键词”组合生成公开搜索查询
5. 页面筛选新增省份筛选和直接搜索
6. 任务页显示最近抓取进度、失败链接样本和重试记录
7. 地图、官网、公众号适配器引入更多 JSON-LD、meta 和正文候选值
8. 增加正式迁移脚本，并在 README 中补环境变量和迁移说明

延续的旧思路：

- 仍然保持 10 个核心业务字段不扩张
- 仍然以规则法为主，不把 AI 判断作为主流程
- 仍然优先围绕公开商务线索，不引入任何隐私采集

修正的旧判断：

- 上一轮虽然已经切到 PostgreSQL，但默认仍偏向“城市级演示任务”
- 本轮明确把“省级任务”和“大批量任务”提前到当前阶段，而不是继续后置

## 7. 本轮优点

1. 任务模型已经从单纯“城市级 demo”推进到支持省级任务
2. 计划抓取上限提高到 `5000`，更接近真实使用场景
3. 任务执行过程不再是黑盒，最近进度、失败样本和重试记录都可视化了
4. 结果页和线索库页支持按省筛选和直接搜索，查找效率更高
5. 来源解析器对品牌、地址、联系方式的抽取比上一轮更细
6. 数据库迁移与环境变量说明更完整，后续接手成本更低

改进的质量属性：

- 可观察性
- 可维护性
- 可扩展性
- 诊断便利性

## 8. 本轮缺点与代价

1. 任务执行器复杂度继续上升，后续如果任务并发增加，进程内队列模式可能需要重构
2. 候选页发现仍主要依赖公开搜索结果页，稳定性依然是 MVP 级
3. 解析器虽然更细了，但仍属于规则增强，不是完整的平台专用解析器
4. 页面层已经展示了更多诊断信息，但还没有做到“按任务查看详细失败历史”的独立详情页
5. 当前直接搜索是在服务端对查询结果做内存过滤，数据量再上去后需要考虑下沉到 SQL 层

当前接受的妥协：

- 先把省级任务和任务诊断跑通，再考虑任务详情页与队列拆分
- 先保持现有 SSR 页面结构，不引入前后端分离改造

## 10. 本轮结论

本轮最重要的结论是：

- 当前系统已经从“PostgreSQL 基线版”进一步进入“支持省级范围、具备任务诊断能力的可持续迭代版”

被证实的假设：

- 任务模型中加入省级字段不会破坏现有页面链路
- 任务页直接展示进度、失败样本和重试记录，对审核与排障都有明显帮助
- 直接搜索和省级筛选能显著降低结果查找成本

需要继续保留的经验：

- 任何与抓取质量相关的增强，最好同时落到“数据模型 + 页面可见性 + 文档说明”三层
- 数据库迁移脚本必须尽早建立，不应长期只依赖一份全量 SQL
