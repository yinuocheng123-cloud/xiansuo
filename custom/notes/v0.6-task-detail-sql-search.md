# 版本记录：v0.6-task-detail-sql-search

## 1. 本轮目标

本轮目标是在 `v0.5-province-progress-migrations` 的基础上，继续把系统往“真实可用的后台工具”推进，重点解决三类问题：

1. 给任务做独立详情页，把失败样本、重试记录和当前候选页状态展开
2. 把结果筛选里的直接搜索下沉到 SQL
3. 继续细化地图、官网、公众号解析器，重点提升品牌归属和地址精度

为什么这轮值得优先做：

- 上一轮已经明确把“任务详情页”列为收益最高的下一步
- 结果搜索仍在内存中过滤，数据量继续上来后会先成为瓶颈
- 来源适配虽然已拆开，但品牌归属和地址精度仍然是影响审核效率的关键点

本轮目标归类：

- 功能补充
- 可维护性提升
- 提取质量提升

## 2. 背景与上下文

本轮承接记录：

- `custom/notes/v0.5-province-progress-migrations.md`

本次与上一轮的关系：

- 完全延续上一轮“在 PostgreSQL 基线上继续做实用化增强”的思路
- 不是新增业务方向，而是继续把已有任务、筛选和解析能力做细

修改前状态：

- 任务页已经有简版诊断信息
- 结果页已经支持省份筛选和直接搜索
- 但任务诊断仍挤在任务列表中，展开深度不够
- 直接搜索仍在服务端取全量结果后做内存过滤
- 品牌和地址的规则虽已增强，但仍偏向“先命中、后人工改”

已知约束：

- 继续严格围绕“高定木作公开商务线索”
- 不突破公开商务信息边界
- 仍以规则法为主，不引入复杂 AI 主流程

## 3. 问题分析

表面现象：

- 任务列表能看到进度，但想排查一个具体任务时仍然不够直观
- 结果页直接搜索虽然能用，但实现方式不适合继续放大数据量
- 审核阶段仍然会遇到“品牌像是对的但不够准”“地址命中了但不够完整”的情况

核心矛盾：

- 当前系统已经不是简单演示，而是在向“可持续人工审核后台”演进
- 如果任务详情、搜索性能和字段精度不继续提升，下一步迭代会开始受到明显限制

主要原因：

1. 缺少独立任务详情页，诊断信息只能压缩展示
2. 直接搜索没有下沉到数据库层
3. 解析器的品牌和地址候选值仍不够系统化

## 4. 候选方案比较

### 方案 A：只补任务详情页，不动 SQL 和解析器

核心思路：

- 新增 `/tasks/:id`
- 不改结果搜索，不继续增强提取

优点：

- 改动最小
- 任务诊断体验能立刻提升

缺点：

- 搜索性能问题继续保留
- 解析质量问题继续保留

为什么放弃：

- 用户这轮的诉求是三项并行推进，只做详情页会留下明显半成品状态

### 方案 B：任务详情页、SQL 搜索、解析器增强一起做

核心思路：

- 任务详情页负责承接所有任务级诊断信息
- 结果筛选下沉到 PostgreSQL `ILIKE`
- 来源适配器继续输出更干净、更丰富的品牌和地址候选

优点：

- 用户需求一次性贯通
- 页面层、存储层、提取层三条线同步变强
- 后续再做任务详情 API 或 SQL 优化时不需要推翻当前结构

缺点：

- 改动跨多个核心文件
- 需要重新核对页面和 API

为什么选中：

- 这轮的三项需求彼此相关，分开做会造成重复返工

## 5. 最终决策

最终采用方案 B，并落实为以下动作：

1. 新增任务详情页 `/tasks/:id`，以及任务详情 API `/api/tasks/:id`
2. 结果列表查询改为 SQL 条件拼装，`q` 直接搜索下沉到 PostgreSQL
3. 任务列表页保留简版诊断，详情页承接完整诊断展开
4. 地图、官网、公众号解析器增加更多标签型候选值和 JSON-LD 候选值
5. `extractor` 对品牌归属和地址做统一清洗、过滤和取舍

延续的旧思路：

- 仍然保持 SSR 后台结构，不引入新的前端框架
- 仍然保持规则优先，不额外扩展字段

修正的旧判断：

- 上一轮直接搜索虽然可用，但本轮明确把它提前下沉到 SQL，而不是再拖一轮

## 6. 具体实现

主要改动文件如下。

### 存储与查询

- `src/lib/storage.js`
  新增 SQL 条件拼装，支持 `taskId`、`reviewStatus`、`province`、`city`、`brand`、`sourcePlatform`、`leadType`、`minScore`、`q`、`limit`
- `src/lib/taskRunner.js`
  `listLeads` 改为直接复用 PostgreSQL 查询，不再自己做内存过滤

### 页面与服务

- `src/lib/render.js`
  新增 `renderTaskDetailPage`
  任务列表改为“简版诊断 + 详情入口”
  任务详情页展开：
  - 当前候选页状态
  - 失败链接样本明细
  - 重试记录明细
  - 该任务最近线索
- `src/server.js`
  新增：
  - `GET /tasks/:id`
  - `GET /api/tasks/:id`

### 提取逻辑

- `src/lib/providers/pageAdapters.js`
  对地图、官网、公众号分别增加更多名称、品牌、地址候选来源
  引入更多标签型文本抽取，例如：
  - 品牌名称
  - 所属品牌
  - 商户名称
  - 展厅地址
  - 门店地址
  - 导航地址
- `src/lib/extractor.js`
  增加品牌候选清洗、泛词过滤、域名品牌回退和地址清洗
  同时把正文摘要长度适当拉长，以提升地址和品牌命中概率

### 文档与记录

- `README.md`
  补任务详情页和 SQL 搜索说明
- `AGENTS.md`
  同步更新最新记录路径

## 7. 本轮优点

1. 任务诊断从“列表中压缩展示”升级到“详情页完整展开”
2. 结果页和线索库页的直接搜索已经下沉到 PostgreSQL，更适合后续扩大数据量
3. 品牌归属和地址提取逻辑更集中，不再完全依赖单条正则碰运气
4. 解析器对地图、官网、公众号的差异化理解更强
5. 任务详情 API 已经具备，为后续接前端交互或独立详情面板打下基础

改善的质量属性：

- 可观察性
- 可维护性
- 提取精度
- 数据查询效率

## 8. 本轮缺点与代价

1. `render.js` 的职责继续变重，后续如果页面继续扩张，可能需要拆分模板模块
2. SQL 筛选当前仍然以 `ILIKE` 为主，数据量非常大时可能还需要全文检索或索引优化
3. 地址清洗和品牌过滤虽然更稳了，但仍有规则误判空间
4. 当前任务详情页只展示最近线索，没有完整任务时间线

当前接受的妥协：

- 先把任务详情、SQL 搜索和规则增强打通，再考虑模板拆分和全文检索

## 10. 本轮结论

本轮最重要的结论是：

- 当前系统已经从“有任务诊断信息”升级到“有独立任务详情视图，并且搜索和提取质量都继续前移优化”的阶段

被证实的假设：

- 任务详情页可以在不改数据库结构的情况下，直接承接已有诊断字段
- 直接搜索下沉到 SQL 不会破坏现有筛选体验
- 来源适配器和 `extractor` 分层增强，是继续提精度的更稳路径

需要继续继承的经验：

- 一旦页面需要展示结构化诊断信息，优先让数据库字段和 API 先稳定，再做前端展开
- 提取精度的增强应尽量集中在适配器和统一清洗层，而不是到处散写特例
